# 進階計算功能 — 游標值模式 (Cursor Mode)

> 狀態：**設計階段（未實作）**
> 適用版本：v6 (b75e390) 之後
> 參考來源：其他工業監控軟體（如 A&D AD-1687 / Onset HOBOware）的「即時/游標」切換概念

---

## 一、功能目標

提供兩種「最新讀值」資料來源切換：

| 模式 | 觸發方式 | 最新讀值表格內容 | 圖表行為 |
|------|----------|------------------|----------|
| **即時狀態** (Live) | 預設 | 顯示最新讀取的即時溫度 | 圖表持續更新最新資料點 |
| **量測狀態** (Cursor) | 切換按鈕 | 顯示「游標區間」內的 **平均 / 最大 / 最小** | 圖表出現 **左右兩條可拖曳垂直線**（x-bar），定義選取區間 |

切換時不改變資料本身（資料還是在持續寫入），只改變「最新讀值表格的計算來源」。

## 三、演算法（後端計算 or 前端計算？）

### 3.1 評估兩個方案

| 方案 | 優點 | 缺點 |
|------|------|------|
| **A. 前端計算** | 切換/拖曳零延遲、不用 call API | 需把整段時間序列推到前端；大量資料時傳輸成本高 |
| **B. 後端 API** | 只傳彙總結果、省頻寬 | 拖曳時每次都要打 API，會有 200~500ms 延遲 |

### 3.2 採用：**前端計算**（推薦方案 A）

理由：
- 目前圖表已經把選定 X 軸範圍的時間序列推到前端（用 LTTB 取樣後的版本）
- LTTB 取樣後的資料點對「平均/最大/最小」已經足夠準確（取樣間距均勻，不會偏誤）
- 拖曳游標是即時互動，前端計算可達 60fps
- 後端維持現有 ring buffer 邏輯不動

### 3.3 計算細節

從前端 chart 拿到的時間序列（已是 LTTB 取樣後）：
```js
const samples = chart.data.datasets[i].data;  // [{x: ts, y: temp}, ...]
```

過濾區間：
```js
function filterRange(samples, tsLeft, tsRight) {
  return samples.filter(s => s.x >= tsLeft && s.x <= tsRight);
}
```

彙總（依大大已確認的演算法原則）：
```js
function summarize(samples) {
  if (samples.length === 0) return { avg: null, max: null, min: null, count: 0 };
  const vals = samples.map(s => s.y).filter(v => v !== null);
  if (vals.length === 0) return { avg: null, max: null, min: null, count: 0 };
  return {
    avg: sum(vals) / vals.length,  // ✓ 分母 = 實際筆數（沿用 v6 既有原則）
    max: Math.max(...vals),
    min: Math.min(...vals),
    count: vals.length,
  };
}
```

**重要：不對「實際筆數 < 理論筆數」做警示，因為：**
- LTTB 取樣後的筆數本來就遠少於原始筆數
- 區間可能跨越 LTTB 取樣空洞（沒有取樣點）
- 對「量測模式」而言，使用者拖曳出來的區間就是「他選的範圍」，不該再加警告

### 3.4 斷線落差計算（依使用者決策 #3）

**需求：考慮期間有斷線，筆數與每 10 秒 1 筆有落差時，要能看出來。**

GX20 預設輪詢週期 = 10 秒 / 筆（見 `config/settings.json` `poll_interval_sec`）。若區間 = 1 小時，理論完整資料 = 360 筆。

#### 方案 A：前端用 LTTB 取樣後筆數推算（推薦）

```
覆蓋率 = 區間內 LTTB 筆數 / 區間秒數 × 10
```

**優點**：
- 不用 API call，拖曳即更新
- LTTB 取樣演算法是「均勻分佈」取樣，理論上覆蓋率反推可接受

**缺點**：
- LTTB 取樣點數與原始資料點數的比例會受圖表 X 軸範圍影響（X 軸拉遠，取樣點變多）
- 準確度比方案 B 差，可能誤差 ±10%

#### 方案 B：後端 API 用 SQLite 計算原始資料覆蓋率

新增 API：`GET /api/cursor/coverage?station=...&t1=...&t2=...`

```python
@app.get("/api/cursor/coverage")
def cursor_coverage(station: str, t1: str, t2: str):
    dt1 = parse(t1)
    dt2 = parse(t2)
    interval_sec = int(s["poll_interval_sec"])  # 預設 10
    actual = query_count(station, dt1, dt2)     # SQLite COUNT(*)
    expected = int((dt2 - dt1).total_seconds() / interval_sec)
    return {"actual": actual, "expected": expected, "pct": actual/expected*100}
```

**優點**：
- 用 SQLite 原始資料，**完全精準**，可正確反映斷線
- 與 v6 既有 `query_recent` 邏輯一致

**缺點**：
- 每次拖曳都要打 API（200~500ms 延遲）— 需 debounce 300ms
- 多一個 endpoint、多一點後端負擔

#### 採用方案：**B（後端 SQLite 查詢）** ✓

- **理由**：使用者需要精準反映斷線，LTTB 推算誤差 ±10% 不可接受
- **副作用**：每次拖曳都打 API，必須加 debounce
- **debounce 設計**：300ms（拖曳結束後 0.3 秒才打 API，連續拖曳時只送最後一次）

#### B 方案 debounce 實作

```javascript
let coverageTimer = null;
function requestCoverage(station, tsLeft, tsRight) {
  clearTimeout(coverageTimer);
  coverageTimer = setTimeout(async () => {
    const resp = await fetch(
      `/api/cursor/coverage?station=${station}&t1=${tsLeft.toISOString()}&t2=${tsRight.toISOString()}`
    );
    const data = await resp.json();
    updateCoverageDisplay(data.actual, data.expected, data.pct);
  }, 300);
}
```

- 拖曳中：游標線即時移動（不需打 API，本地計算平均/最大/最小）
- 拖曳停止 0.3 秒後：打 API 更新覆蓋率
- 表格上的「平均/最大/最小」純前端計算，**始終即時**（不 debounce）
- 只有「資料覆蓋」那一行會延遲 0.3 秒，使用者體感幾乎無感

## 四、Chart.js 整合設計

### 4.1 現有圖表結構

`main.js` 用 Chart.js v4，多 dataset（每工位一條線）。需要在 chart options 加入 annotation plugin：

```js
// chart config
{
  options: {
    plugins: {
      annotation: {
        annotations: {
          cursorLeft: { type: 'line', xMin: ..., xMax: ..., borderColor: '#28a745', borderDash: [6,6] },
          cursorRight: { type: 'line', xMin: ..., xMax: ..., borderColor: '#dc3545', borderDash: [6,6] },
          cursorRange: { type: 'box', xMin: ..., xMax: ..., backgroundColor: 'rgba(255,235,59,0.12)' }
        }
      }
    }
  }
}
```

### 4.2 拖曳互動

用 chartjs-plugin-dragdata（已存在於 v6 依賴或需新增）：

- 兩條 annotation line 各加 `draggable: true`
- onDrag 回呼：更新 `cursorLeft.value` / `cursorRight.value`、重繪 chart、重新計算表格

### 4.3 邊界處理

- 拖曳時自動夾住：左 ≤ 右（否則交換或忽略）
- 拖曳時自動夾住 X 軸範圍（不讓游標跑到圖表外）
- 兩游標最小距離 = 圖表寬度的 1%（避免兩線完全重疊）

### 6.2 邊界測試

- 區間內無資料 → 表格顯示 — (None)
- 區間內只有 1 筆 → 平均/最大/最小都等於該值
- 區間跨越圖表空洞（LTTB 取樣點稀疏）→ 用實際落在區間內的點算
- 跨午夜資料 → 時間戳記正確比較

### 6.3 回歸測試

- v6 既有功能（CSV 匯出、清除此工位、設定頁、暗黑主題）全部不受影響
- avg 公式仍是 `sum/len`，不因新模式而改

## 八、實作決策記錄

### 決策 1：前端計算 vs 後端 API
- **選擇**：前端計算
- **理由**：拖曳需即時回應；LTTB 取樣已提供足夠精度
- **日期**：2026-06-12

### 決策 2：游標線顏色
- **選擇**：綠/紅對比（沿用參考軟體）
- **理由**：工業監控軟體慣例，辨識度高
- **日期**：2026-06-12

### 決策 3：拖曳即更新 vs 按計算按鈕
- **選擇**：拖曳即更新
- **理由**：參考軟體需按「計算」按鈕，但現代 web 拖曳即時更新更直覺；零延遲體驗優先
- **日期**：2026-06-12

### 決策 4：游標位置是否跨工位保留
- **選擇**：**不保留**，切換工位時強制回到即時狀態（游標線隱藏、表格切回即時欄位）
- **理由**：使用者決策 #2「切換工站時以即時狀態開始，不用考慮上一個畫面狀態」
- **日期**：2026-06-12（修訂）

### 決策 5：游標線可見性
- **選擇**：游標線、區間 highlight、區間資訊列**只在量測狀態顯示**；即時狀態全部隱藏
- **理由**：使用者決策 #5「游標線只在切換為游標狀態時顯示」
- **日期**：2026-06-12

### 決策 6：斷線覆蓋率計算方式
- **選擇**：**B（後端 SQLite 查詢）**
- **理由**：使用者決策 #3 要求精準反映斷線；LTTB 推算誤差 ±10% 不可接受
- **副作用與隱藏**：拖曳中表格平均/最大/最小即時更新，覆蓋率用 300ms debounce 延遲更新
- **日期**：2026-06-12

### 決策 7：是否保留「計算」按鈕
- **選擇**：**取消**
- **理由**：使用者決策 #1「若可即時計算，取消計算鍵」；拖曳即時更新，無需按鈕觸發
- **日期**：2026-06-12
